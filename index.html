Current status & milestone

Current milestone

The core day-to-day usage loop is feature-complete: a user can sign up for a free trial, create a 2D card, edit it, set it active, share it (QR, message, email, WhatsApp/Telegram, native share sheet), view it full-screen, manage their team under Users, build a saved-contacts address book by scanning a QR code, send a push-notification announcement to everyone who has their card, receive one back too (a session-bootstrap banner plus a full screen reached by tapping the push), and — as of this update — permanently delete their account, all talking to the real production backend with unchanged request formats.

Next milestone

Not yet decided. Remaining candidates: 360° cards, Bluetooth P2P sharing, Unity's separate already-paid-tenant Store surface, forgot-password, and Tutorials/onboarding — see the Planned rows below.

Risks & blockers

  • A handful of backend quirks (e.g. a blob-path mismatch in the card-layout-edit endpoint) are being faithfully reproduced rather than fixed, per plan — a dedicated backend clean-up pass is scheduled for after full UI parity, not before.
  • 360° cards and Bluetooth P2P sharing need native plugin integration (panoramic viewer, BLE GATT protocol) — larger, riskier scope than the screens shipped so far.
  • The paid-subscription purchase flow (both iOS StoreKit and Android Play Billing) is wired for release builds, but neither store has a real product catalog configured yet (App Store Connect / Play Console), so real purchases still fall through to the simulated debug flow in practice until that catalog work happens; server-side receipt verification stays deliberately deferred (see ADR 0001).
  • Push notifications are fully wired (OneSignal init, permission request, click routing, session identity linking) and unit/widget-tested, but real delivery hasn't been verified on a device — Android needs an FCM sender configured in the OneSignal dashboard, iOS needs the Push Notifications capability + an APNs key, neither of which this session had console access to do (see ADR 0002). Both platforms also still ship under the Flutter template's placeholder application id, same pre-existing gap ADR 0001 already flagged for Play Billing.

Recent updates

  • 21 Jul 2026

    Account deletion

    Reached from the account menu's "Delete Account" row (previously a dummy): typing the literal word "DELETE" is the client-side confirmation gate, exactly mirroring Unity's `VBC_Handler.DeleteAccount()` input check, before `vbc/deleteaccountfromapp` is called at all. Mirrors Unity's two-step success flow rather than an instant pop — a confirmed deletion clears local storage immediately but keeps the in-memory session alive long enough to show a success view, only clearing it (and letting the router's redirect guard bounce to login) once the user taps through, matching `AccountDeletionConfirmed()` → `FinishDeletion()`. New tests cover the repository/controller success and failure paths and the screen's validation and success states; full suite is 225 tests, all passing. Live-verified against the real test backend for rendering and the type-DELETE validation gate (both empty and incorrect-case input correctly blocked client-side); the actual destructive delete call was deliberately not exercised against the shared test account, to keep it available for future QA.

  • 21 Jul 2026

    Receiving announcements & push notifications

    Two surfaces, mirroring two genuinely different Unity mechanisms: a session-bootstrap banner on Home (VBC_Handler's `announcements` field from vbc/getdatawithemail — shown unconditionally on every login/refresh, since there's no dismiss/mark-read endpoint, faithfully reproducing that quirk), and a full received-announcement screen reached by tapping a push notification (message, sender, optional image, "Open sender's card" looking up the saved contact by external_id). OneSignal (`onesignal_flutter`) is wired end to end: SDK init + permission request + click listener at app start, and session-identity linking (login/logout paired with the tenant uuid, deduped so a same-tenant session refresh doesn't re-link) kept as a separate `ref.listen`-based provider rather than folded into AuthController, so the other ~200 tests that merely read the session never touch a real platform channel. Reuses Unity's exact OneSignal App ID so pushes keep reaching existing production users. Real push delivery needs Android/iOS console-side setup (FCM sender, APNs key) this session couldn't do — see the new ADR 0002. 21 new tests (payload parsing incl. malformed/missing-field safety, the banner controller's re-show-on-every-refresh quirk, the identity-sync dedup logic, the received screen's sender lookup) — full suite is 216 tests, all passing.

  • 21 Jul 2026

    Real Android IAP (Play Billing) for paid signup

    Extended the Subscribe purchase flow (signup plan screen + Store trial-conversion) to real Google Play Billing in Android release builds, alongside iOS StoreKit — both now go through the same RealIapService, since package:in_app_purchase abstracts the two stores identically; no store-specific code was needed. Android debug/profile builds keep the simulated flow, since Play Billing has no fully-local test path the way StoreKit Testing does. Matches Unity's IAPManager precedent of not sending a receipt to the backend at signup time on either platform. ADR 0001 updated to reflect Android is no longer deferred; real purchases on either store still need their respective consoles' product catalogs configured (and the app's still-placeholder applicationId fixed) before they'll actually work outside the debug flow.
