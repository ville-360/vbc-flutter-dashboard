Current status & milestone

Current milestone

The core day-to-day usage loop is feature-complete: a user can sign up for a free trial, create a 2D card, edit it (with live upload/processing progress on the tile — as of this update), set it active, share it (QR, message, email, WhatsApp/Telegram, native share sheet, and now real Bluetooth P2P — see below), view it full-screen (including a saved contact's/shared card), manage their team under Users, build a saved-contacts address book by scanning a QR code, send a push-notification announcement to everyone who has their card, receive one back too (a session-bootstrap banner plus a full screen reached by tapping the push), permanently delete their account, and recover a forgotten password, all talking to the real production backend with unchanged request formats. Authentication & Onboarding and Card Creation & Management are now both fully done.

Next milestone

Not yet decided. Remaining candidates: Unity's separate already-paid-tenant Store surface and Tutorials/onboarding — see the Planned rows below. 360° cards have been moved to the Backlog tab and aren't part of the near-term roadmap.

Risks & blockers

  • A handful of backend quirks (e.g. a blob-path mismatch in the card-layout-edit endpoint) are being faithfully reproduced rather than fixed, per plan — a dedicated backend clean-up pass is scheduled for after full UI parity, not before.
  • Bluetooth P2P sharing (nearby list, tap-to-share, accept/decline) is built and unit/widget-tested against a fake BLE service, but the real native transport (`bluetooth_low_energy`'s Android/iOS platform channels) can't be exercised in this environment — no BLE radio, and neither the iOS Simulator nor most Android emulators support real BLE peripheral/central operation. A follow-up pass with two physical devices is needed to confirm real-world pairing. 360° cards would need similar native-plugin work (a panoramic viewer) but have been deprioritized to the Backlog tab.
  • The app now ships under its real, live production identifier (com.Mediahouse.VBCapp, both Android and iOS — confirmed directly from Unity's own project settings) instead of the Flutter template placeholder, since the plan is for this app to replace Unity's as an update to the same store listings, not launch separately. This meaningfully de-risks the three items below — Play Console/App Store Connect's product catalogs, OneSignal's Android FCM sender, and the domain's App/Universal Link association files should all already exist for this exact identity from the live Unity app — but none of it has been confirmed against a real signed build yet, and Android specifically still needs the release build **signed with the same keystore** Unity ships with (matching applicationId alone isn't sufficient for Google Play to accept an update).
  • The paid-subscription purchase flow (both iOS StoreKit and Android Play Billing) is wired for release builds; with the real applicationId now in place, the existing product catalogs should apply without new console setup, but that's unconfirmed from this environment. Server-side receipt verification is tracked separately in the Backlog tab (see ADR 0001).
  • Push notifications are fully wired (OneSignal init, permission request, click routing, session identity linking) and unit/widget-tested — the receiving UI (banner + full screen) is done — but real delivery hasn't been verified on a device. Android's FCM sender is likely already configured (same OneSignal App ID + real package name the live Unity app already uses); iOS still needs a one-time "Push Notifications" Xcode capability pass plus confirming an APNs key exists for this Team/bundle id (see ADR 0002).
  • Widget & deep-link sharing's app-side logic is code-complete and tested, including a real bug caught before shipping (the Android widget's `vbc_app://` scheme has an underscore, invalid per RFC 3986 — `package:app_links`' URI-based stream silently drops it, so the raw-string stream is used instead). The domain almost certainly already hosts the Android/iOS association files referencing this identity, since Unity's live app already handles these same links — but the iOS Associated Domains entitlement still needs a one-time Xcode capability pass to actually wire into the build (see ADR 0003).

  • 22 Jul 2026

    Real applicationId/Bundle ID (com.Mediahouse.VBCapp)

    The plan is for this Flutter app to replace the Unity app as an update to its existing App Store/Play Store listings, not ship as a separate app — which means it has to use the exact same identity Unity ships under. Confirmed that identifier directly from Unity's own `ProjectSettings.asset` (`com.Mediahouse.VBCapp`, identical string for both platforms) rather than assuming it, and set it in `android/app/build.gradle.kts`'s `applicationId` and every `PRODUCT_BUNDLE_IDENTIFIER` in the iOS Xcode project (Runner + RunnerTests, all build configurations). Left the internal Kotlin namespace as the Flutter template default deliberately — Android explicitly supports `namespace != applicationId`, and changing it would mean renaming source files/directories for zero store-facing or functional benefit. This meaningfully de-risks the identity-shaped gaps ADR 0001/0002/0003 already flagged for IAP, push notifications, and deep linking — Play Console/App Store Connect's product catalogs, OneSignal's Android FCM sender, and the domain's Universal/App Link association files should all already exist for this exact identity from the live Unity app, since none of that infrastructure cares which technology built the binary. All three ADRs updated to reflect this; none of it has been confirmed against a real signed build from this environment, and Android specifically still needs release builds signed with Unity's actual keystore (`mediatalokey.keystore`) — matching applicationId alone isn't sufficient for Google Play to accept an update to the existing listing. Full test suite (323 tests) and `flutter analyze` still clean after the change; a real `flutter build web` compile also still succeeds.

  • 22 Jul 2026

    Widget & deep-link sharing

    The app-side response to a shared-card link or a tap on the old native home-screen widget (not the widget itself — see the dashboard-correction entry below). A shared-card link (view./card.virtualbusinesscard.ai with a guestpass) now opens straight to that card via the existing `SharedCardViewerScreen`, even with no session at all — the router's redirect guard now exempts that one route, matching `oculus/gettraditionalcarddata` being a genuinely public endpoint the same way Unity's own `OpenCardFromOutside()` needs no login. A widget tap opens Share. Caught a real bug before it could ship silently: the Android widget's actual shipped code (`VBCAppWidget.java`) uses the scheme `vbc_app://`, which contains an underscore — invalid per RFC 3986 — so Dart's strict `Uri.parse` rejects it, and `package:app_links`' URI-based stream resolves links via `Uri.tryParse` internally and would have silently dropped the widget's tap with no error at all. Fixed by parsing the raw string `app_links` also exposes (`stringLinkStream`) instead, with lenient prefix-checks ahead of any real URI parsing. Real device delivery of the Universal/App Links path (not the widget-tap path, which needs no domain hosting) still needs domain-hosted association files and the app's real bundle id — full writeup in the new `docs/adr/0003`. 12 new tests (link parsing across both schemes and both hosts, the router's new public-route exemption across every auth state) — full suite is 323 tests, all passing, plus a clean `flutter analyze` and a real `flutter build web` compile check.

    Dev/QA server switch panel; HEIC image conversion

    Two small platform-hardening pieces, both faithful-to-Unity in mechanism, idiomatic Flutter in implementation. Dev/QA panel: ten rapid taps on the login screen's version label — the same combo timing as `DevSettings.cs`'s `OnButtonTapped()` (a reusable `RapidTapDetector` rather than one-off tap-counting state) — opens a bottom sheet to switch between Default/Test env/Localhost, immediately rebuilding the shared `Dio` client (and everything built on it) via Riverpod's dependency graph instead of requiring a restart. HEIC conversion: a real, live gap, not just a historical one — `image_picker` does not normalize HEIC away on iOS, and this app's blob paths/content-type headers for card media are hardcoded to `image/png` (matching Unity's own upload convention), so an unconverted HEIC pick would silently upload a corrupt image. Every image pick (card main media, logo, announcements) now runs through one shared conversion gate in `MediaPickerService` (`flutter_image_compress`, format: PNG) instead of Unity's several separate `HeicConverter.Convert()` call sites, with the exact same graceful fallback Unity's own `#else` branch has — a failed/unsupported conversion (e.g. web, which can't decode HEIC input via Canvas at all) passes the original bytes through rather than blocking the pick. 27 new tests (the tap-detector's combo/reset/threshold logic, environment persistence and Dio-rebuild-on-switch, the dev sheet's UI states, and HEIC detection/conversion/fallback against a fake converter) — full suite is 309 tests, all passing, plus a clean `flutter analyze` and a real `flutter build web` compile check.

  • 22 Jul 2026

    Bluetooth P2P card sharing

    A real nearby-list + tap-to-share + accept/decline flow on the Share screen, replacing the disabled "Bluetooth: Soon" tile — deliberately not a port of `BluetoothManager.cs`'s actual shipped mechanics (every device auto-connects to and auto-blasts its card at any other nearby device the instant a BLE connection forms, with no real list or user intent involved — the tap-to-send UI exists in Unity's code but the call that would wire it up is dead). Unity was used only to confirm the feature's goal and its two backend calls. New `bluetooth_low_energy` dependency (a single package covering both central/scanning and peripheral/advertising roles, verified live against pub.dev before committing to it) powers a from-scratch GATT protocol: a fixed custom service UUID for discovery, a truncated advertised name for the nearby-list label (BLE ad packets are only a few dozen bytes), and a `share_request` characteristic carrying the full untruncated identity as JSON once a user actually taps someone — no reply/ack, matching Unity's own asymmetry that the sender never learns the outcome. On Accept: `vbc/vbccontactaccepted` (exact Unity field set) resolves the full card, `vbc/savecontact` persists it under Contacts (reusing the existing repository verbatim), then the sender's card opens via the just-shipped `SharedCardViewerScreen`. Advertising/scanning/listening are all tied to the Share screen's own lifecycle (start on mount, stop on unmount) — a request only works while both people currently have Share open, the same scope Unity's own mechanism needed, called out explicitly as a disclosed decision rather than a silent one. 36 new tests (protocol encode/decode incl. malformed-payload and UTF-8-safe truncation, the nearby-list and incoming-request controllers against a fake BLE service, both new widgets across every state, and a full Share-screen accept/decline/navigate integration test) — full suite is 282 tests, all passing, plus a clean `flutter analyze` and a real `flutter build web` compile check. The native BLE transport itself (real advertising/scanning/pairing on physical Android/iOS hardware) can't be verified in this environment — no BLE radio, and neither the iOS Simulator nor most Android emulators support real BLE peripheral/central operation — flagged as a follow-up needing two physical devices, same accepted boundary as this app's existing IAP/push-notification work.
